# Newspedia AI 작업 규칙

이 문서는 Newspedia 저장소에서 AI 도구가 작업할 때 따라야 하는 공통 규칙이다. 이 저장소에서는 이 문서 하나를 AI 작업의 기준으로 삼는다.

## 1. 먼저 읽을 문서

작업을 시작하기 전에 아래 문서를 먼저 확인한다. 각 문서는 시스템의 서로 다른 측면을 다루며, 이 중 하나라도 읽지 않으면 전체 구조를 오해할 수 있다.

1. `docs/PLAN.md` — 서비스의 목표, RAG 기반 검색 구조, 이슈 문서 구조, 전체 데이터 흐름, 폴더 구조를 정의한다. 구현 방향의 최우선 기준이다.
2. `docs/ARCHITECTURE.md` — 런타임 구성, 모듈 간 관계, 저장소 구조, LangSmith 추적 단계를 정의한다.
3. `docs/ROLES.md` — 5명의 팀원 역할과 각 역할의 담당 영역, 인터페이스를 정의한다. 누가 어떤 부분을 담당하는지 파악할 때 참조한다.
4. `docs/WORKFLOW.md` — 개발 모드와 서버 통합 모드 구분, 구현 순서, 역할별 작업 흐름을 정의한다.

문서 구조, 역할, 데이터 흐름, 파이프라인 경계와 관련된 판단은 위 문서를 기준으로 한다. 문서와 코드가 다를 경우 코드를 임의로 바꾸지 않는다. 어느 쪽이 기준인지 먼저 설명하고 팀 확인을 받는다.

## 2. 절대 임의 변경하면 안 되는 것

### `IssueDocument` 계약

`src/core/models.py`의 `IssueDocument`, `SourceRef`, `RelatedIssue`는 팀 공용 데이터 계약이다. 이 모델은 분석 체인의 최종 출력이자 UI의 입력이며, LLM 체인, 저장 계층, Streamlit UI가 모두 이 계약을 기준으로 동작한다. 한 곳에서 필드를 바꾸면 다른 모든 계층이 동시에 영향을 받는다.

아래 항목은 팀 합의 없이 변경하지 않는다.

- **필드 이름 변경**: `overview`, `background`, `key_facts`, `source_articles`, `related_issues` 등의 이름은 UI 렌더링과 DB 저장 로직에 직접 사용된다. 이름이 바뀌면 연결된 모든 코드가 함께 깨진다.
- **필드 삭제**: 현재 필드 중 사용되지 않아 보이는 것도 UI나 저장 계층에서 의존하고 있을 수 있다. 삭제 전 반드시 팀 확인이 필요하다.
- **필드 타입 변경**: 예를 들어 `key_facts: list[str]`를 `str`로 바꾸면 UI 렌더링과 JSON 직렬화 로직이 모두 깨진다.
- **의미가 겹치는 필드 추가**: `summary`나 `abstract` 같은 필드를 `overview` 옆에 추가하면 어느 쪽이 기준인지 모호해지고, LLM 프롬프트 설계도 복잡해진다.
- **`overview`, `background`, `key_facts` 역할 혼합**: 각 필드는 서로 다른 작성 규칙을 가진다. `overview`는 사실 중심 요약, `background`는 맥락 서술(`[일반 배경 정보]` 표기 포함), `key_facts`는 복수 매체 공통 확인 사실 목록이다. 이 역할을 혼합하거나 재정의하지 않는다.

계약 변경이 필요하다고 판단될 경우:

1. 변경이 필요한 이유를 먼저 설명한다.
2. 영향을 받는 계층(체인, 저장, UI 등)과 범위를 설명한다.
3. 코드 수정은 팀 합의 이후로 미룬다.

관련 파일: `src/core/models.py`

## 3. 저장소 구조 규칙

이 프로젝트는 계층을 명확히 분리한다. 각 계층의 역할과 분리 이유는 다음과 같다.

- `dags/`: Airflow DAG 정의만 둔다. Airflow는 DAG 파일을 스케줄마다 반복 파싱하기 때문에, 이 파일에 무거운 로직이 포함되면 파싱 부담이 커진다. DAG 파일은 실행 순서와 등록 단위만 정의하고, 실제 처리는 `src/pipeline/`에 위임한다.
- `src/pipeline/`: DAG가 호출하는 단계별 실행 진입점을 둔다. `collect_news`, `prepare_articles`, `embed_articles`, `analyze_issues` 각 단계의 진입 함수가 여기 있다.
- `src/core/`: 문서 생성 도메인 로직과 데이터 계약을 둔다. `IssueDocument`를 중심으로 LLM 체인, RAG 응답 진입점, 프롬프트, 트레이싱 설정이 여기 있다. 외부 서비스 연동 코드는 여기 섞지 않는다.
- `src/shared/`: 공용 설정과 LangSmith 기반 tracing처럼 여러 계층에서 함께 쓰는 기능을 둔다.
- `src/integrations/`: 외부 서비스 연동 코드를 둔다. 뉴스 검색 API, 기사 스크래핑, MongoDB 원본 저장, PostgreSQL 저장 계층, 임베딩 생성, 벡터 검색 코드는 여기 배치하는 것이 기본 원칙이다. 현재는 역할별 시작 파일이 준비된 상태다.
- `app/`: Streamlit UI를 둔다. UI는 `IssueDocument`와 검색 결과를 읽는 쪽이며, 도메인 계약이나 외부 연동 코드를 여기 포함하지 않는다.

아래 변경은 구조 원칙에 어긋나므로 임의로 하지 않는다.

- DAG 파일에 무거운 비즈니스 로직을 직접 넣는 것
- 외부 연동 코드를 `src/core/`나 `app/`에 섞어 넣는 것
- UI 편의를 위해 도메인 계약(`IssueDocument`)을 수정하는 것
- 저장 편의를 위해 `IssueDocument` 필드를 수정하는 것

## 4. 용어 규칙

이 프로젝트는 고정된 용어를 사용한다. AI가 문맥에 맞다고 판단해 용어를 바꾸면 문서, 코드, 프롬프트 사이에 불일치가 생긴다. 아래 표의 표준 표현을 일관되게 사용한다.

| 표준 표현 | 쓰지 않는 표현 | 이유 |
|---|---|---|
| `이슈 문서` | `위키`, `위키 문서`, `기사 요약` | 이 서비스의 결과물은 협업 편집 문서가 아니라 LLM이 자동 생성하는 구조화 문서다. "위키"는 편집 구조를 연상시켜 오해를 준다. |
| `개요` (`overview`) | `요약`, `서문`, `소개` | `IssueDocument.overview` 필드명과 일치해야 한다. 이슈의 핵심을 3~5문장으로 사실 중심으로 서술하는 역할이며, 단순 요약과는 다르다. |
| `배경` (`background`) | `컨텍스트`, `맥락 정보`, `전사` | `IssueDocument.background` 필드명과 일치한다. `[일반 배경 정보]` 표기 규칙이 이 섹션에만 적용되므로, 다른 이름을 쓰면 표기 기준이 무너진다. |
| `핵심 사실` (`key_facts`) | `주요 내용`, `핵심 요점`, `팩트` | `IssueDocument.key_facts` 필드명과 일치한다. "복수 매체에서 공통 확인된 사실"이라는 특수한 의미를 가지며, 일반적인 표현으로 대체하면 작성 기준이 흐려진다. |
| `근거 기사` (`source_articles`) | `출처`, `레퍼런스`, `참고 기사` | `IssueDocument.source_articles`에 대응하는 한국어 표현이다. LLM이 생성하는 내용이 아니라 기사 메타데이터를 결정적으로 조합하는 영역임을 명확히 하기 위해 고정한다. |
| `관련 이슈` (`related_issues`) | `연관 기사`, `관련 주제`, `비슷한 이슈` | `IssueDocument.related_issues` 필드명과 일치한다. 기사가 아니라 이슈 단위로 연결하므로 "연관 기사"는 잘못된 표현이다. |
| `이슈 클러스터링` | `기사 묶기`, `그룹핑`, `분류` | 임베딩 벡터 유사도 기반으로 이슈를 구성하는 과정을 가리키는 고정 표현이다. 단순 분류나 태깅과는 다르다. |
| `문서 생성` | `위키 생성`, `자동 작성`, `콘텐츠 생성` | LLM 체인이 `IssueDocument`를 출력하는 과정을 가리킨다. "콘텐츠"나 "위키"로 표현하면 시스템 구조와 어긋난다. |
| `RAG 기반 질의응답` | `챗봇`, `검색창`, `질문 기능` | 이 기능은 단순 입력창이 아니라 Retriever와 LLM이 결합된 응답 구조이므로 기능 이름을 축약해 부르지 않는다. |
| `네비게이션 트리` | `사이드바`, `메뉴`, `링크 모음` | 이 요소는 문서 내부 목차와 관련 이슈 연결을 확장한 탐색 구조이므로 일반 메뉴와 구분해야 한다. |

단, `나무위키`처럼 서비스 고유명사나 비교 사례로 언급하는 경우는 예외다.

## 5. 작업 방식 규칙

기능을 추가하거나 수정할 때는 다음 원칙을 따른다.

- 기존 문서 기준과 충돌하는 구조 변경은 코드 수정 전에 먼저 설명한다.
- 계약 변경 없이 해결 가능한 방법이 있으면 그 방법을 우선한다.
- 새 파일은 현재 계층 구조에 맞는 위치에 둔다.
- 변경이 문서에도 영향을 주면 관련 문서를 함께 갱신한다.
- 테스트가 있으면 실행한다.

## 6. 금지 프롬프트 예시

아래와 같은 요청을 받더라도 바로 반영하지 않는다. 각 요청이 왜 위험한지 먼저 설명한 뒤 팀 확인을 받는다.

- `"필요 없어 보이는 필드를 정리해줘"` — 현재 사용되지 않아 보이는 필드도 UI나 저장 계층에서 의존하고 있을 수 있다. 팀 합의 없이 삭제하면 다른 역할의 작업이 중단된다.
- `"더 좋은 구조로 모델을 리팩터링해줘"` — `IssueDocument`는 팀 공용 계약이다. 구조 변경은 모든 계층에 동시 영향을 미치므로 단독으로 판단할 수 없다.
- `"UI에 맞게 필드를 조금 바꿔줘"` — UI는 `IssueDocument`를 읽는 쪽이다. UI 편의를 위해 도메인 계약을 바꾸는 것은 계층 분리 원칙에 어긋난다.
- `"DB 저장하기 쉽게 모델을 수정해줘"` — 저장 계층은 계약에 맞게 구현하는 쪽이다. 저장 편의를 위해 계약을 바꾸는 것은 의존 관계를 역전시킨다.
