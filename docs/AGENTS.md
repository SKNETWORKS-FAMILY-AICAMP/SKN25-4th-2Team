# ArXplore AI 작업 규칙

이 문서는 ArXplore 저장소에서 AI 도구가 작업할 때 따라야 하는 공통 규칙이다. 이 저장소에서는 이 문서 하나를 AI 작업의 기준으로 삼는다.

## 1. 먼저 읽을 문서

작업을 시작하기 전에 아래 문서를 먼저 확인한다. 각 문서는 시스템의 서로 다른 측면을 다루며, 이 중 하나라도 읽지 않으면 전체 구조를 오해할 수 있다.

1. `docs/PLAN.md` — 서비스의 목표, RAG 기반 검색 구조, 토픽 문서 구조, 전체 데이터 흐름, 폴더 구조를 정의한다. 구현 방향의 최우선 기준이다.
2. `docs/ARCHITECTURE.md` — 런타임 구성, 모듈 간 관계, 저장소 구조, LangSmith 추적 단계를 정의한다.
3. `docs/ROLES.md` — 5명의 팀원 역할과 각 역할의 담당 영역, 인터페이스를 정의한다. 누가 어떤 부분을 담당하는지 파악할 때 참조한다.
4. `docs/WORKFLOW.md` — 개발 모드와 서버 통합 모드 구분, 구현 순서, 역할별 작업 흐름을 정의한다.

문서 구조, 역할, 데이터 흐름, 파이프라인 경계와 관련된 판단은 위 문서를 기준으로 한다. 문서와 코드가 다를 경우 코드를 임의로 바꾸지 않는다. 어느 쪽이 기준인지 먼저 설명하고 팀 확인을 받는다.

## 2. 절대 임의 변경하면 안 되는 것

### `TopicDocument` 계약

`src/core/models.py`의 `TopicDocument`, `PaperRef`, `RelatedTopic`는 팀 공용 데이터 계약이다. 이 모델은 분석 체인의 최종 출력이자 UI의 입력이며, LLM 체인, 저장 계층, Streamlit UI가 모두 이 계약을 기준으로 동작한다. 한 곳에서 필드를 바꾸면 다른 모든 계층이 동시에 영향을 받는다.

아래 항목은 팀 합의 없이 변경하지 않는다.

- **필드 이름 변경**: `overview`, `key_findings`, `papers`, `related_topics` 등의 이름은 UI 렌더링과 DB 저장 로직에 직접 사용된다. 이름이 바뀌면 연결된 모든 코드가 함께 깨진다.
- **필드 삭제**: 현재 필드 중 사용되지 않아 보이는 것도 UI나 저장 계층에서 의존하고 있을 수 있다. 삭제 전 반드시 팀 확인이 필요하다.
- **필드 타입 변경**: 예를 들어 `key_findings: list[str]`를 `str`로 바꾸면 UI 렌더링과 JSON 직렬화 로직이 모두 깨진다.
- **의미가 겹치는 필드 추가**: `summary`나 `abstract_summary` 같은 필드를 `overview` 옆에 추가하면 어느 쪽이 기준인지 모호해지고, LLM 프롬프트 설계도 복잡해진다.
- **`overview`, `key_findings` 역할 혼합**: `overview`는 토픽 수준의 흐름 요약이고, `key_findings`는 검증 가능한 연구 기여와 공통 경향 목록이다. 두 역할을 혼합하거나 재정의하지 않는다.

계약 변경이 필요하다고 판단될 경우:

1. 변경이 필요한 이유를 먼저 설명한다.
2. 영향을 받는 계층(체인, 저장, UI 등)과 범위를 설명한다.
3. 코드 수정은 팀 합의 이후로 미룬다.

관련 파일: `src/core/models.py`

## 3. 저장소 구조 규칙

이 프로젝트는 계층을 명확히 분리한다. 각 계층의 역할과 분리 이유는 다음과 같다.

- `dags/`: Airflow DAG 정의만 둔다. Airflow는 DAG 파일을 스케줄마다 반복 파싱하기 때문에, 이 파일에 무거운 로직이 포함되면 파싱 부담이 커진다. DAG 파일은 실행 순서와 등록 단위만 정의하고, 실제 처리는 `src/pipeline/`에 위임한다.
- `src/pipeline/`: DAG가 호출하는 단계별 실행 진입점을 둔다. `collect_papers`, `prepare_papers`, `embed_papers`, `analyze_topics` 각 단계의 진입 함수가 여기 있다.
- `src/core/`: 문서 생성 도메인 로직과 데이터 계약을 둔다. `TopicDocument`를 중심으로 LLM 체인, RAG 응답 진입점, 프롬프트, 트레이싱 설정이 여기 있다. 외부 서비스 연동 코드는 여기 섞지 않는다.
- `src/shared/`: 공용 설정과 LangSmith 기반 tracing처럼 여러 계층에서 함께 쓰는 기능을 둔다.
- `src/integrations/`: 외부 서비스 연동 코드를 둔다. HF Daily Papers 연동, MongoDB 원본 저장, PostgreSQL 저장 계층, 임베딩 생성, 벡터 검색 코드는 여기 배치하는 것이 기본 원칙이다.
- `app/`: Streamlit UI를 둔다. UI는 `TopicDocument`와 검색 결과를 읽는 쪽이며, 도메인 계약이나 외부 연동 코드를 여기 포함하지 않는다.

아래 변경은 구조 원칙에 어긋나므로 임의로 하지 않는다.

- DAG 파일에 무거운 비즈니스 로직을 직접 넣는 것
- 외부 연동 코드를 `src/core/`나 `app/`에 섞어 넣는 것
- UI 편의를 위해 도메인 계약(`TopicDocument`)을 수정하는 것
- 저장 편의를 위해 `TopicDocument` 필드를 수정하는 것

## 4. 용어 규칙

이 프로젝트는 고정된 용어를 사용한다. AI가 문맥에 맞다고 판단해 용어를 바꾸면 문서, 코드, 프롬프트 사이에 불일치가 생긴다. 아래 표의 표준 표현을 일관되게 사용한다.

| 표준 표현 | 쓰지 않는 표현 | 이유 |
|---|---|---|
| `토픽 문서` | `이슈 문서`, `위키`, `논문 요약` | 결과물은 협업 편집 문서가 아니라 LLM이 자동 생성하는 구조화 문서다. |
| `개요` (`overview`) | `요약`, `서문`, `소개` | `TopicDocument.overview` 필드명과 일치해야 한다. 토픽 전체 흐름을 3~5문장으로 설명하는 역할이다. |
| `핵심 발견` (`key_findings`) | `핵심 사실`, `주요 내용`, `핵심 요점` | `TopicDocument.key_findings` 필드명과 일치한다. 공통 연구 기여와 차이를 요약하는 특수한 의미를 가진다. |
| `논문` (`papers`) | `기사`, `출처 기사`, `근거 기사` | 데이터 소스가 뉴스가 아니라 논문이다. |
| `관련 토픽` (`related_topics`) | `관련 이슈`, `연관 기사`, `관련 주제` | 현재 문서는 토픽 단위로 연결된다. |
| `토픽 클러스터링` | `기사 묶기`, `그룹핑`, `분류` | 임베딩 유사도 기반으로 토픽을 구성하는 과정을 가리키는 고정 표현이다. |
| `문서 생성` | `위키 생성`, `자동 작성`, `콘텐츠 생성` | LLM 체인이 `TopicDocument`를 출력하는 과정을 가리킨다. |
| `RAG 기반 질의응답` | `챗봇`, `검색창`, `질문 기능` | 이 기능은 Retriever와 LLM이 결합된 응답 구조다. |

단, `나무위키`처럼 서비스 고유명사나 비교 사례로 언급하는 경우는 예외다.

## 5. 작업 방식 규칙

기능을 추가하거나 수정할 때는 다음 원칙을 따른다.

- 기존 문서 기준과 충돌하는 구조 변경은 코드 수정 전에 먼저 설명한다.
- 계약 변경 없이 해결 가능한 방법이 있으면 그 방법을 우선한다.
- 새 파일은 현재 계층 구조에 맞는 위치에 둔다.
- 변경이 문서에도 영향을 주면 관련 문서를 함께 갱신한다.
- 테스트가 있으면 실행한다.

## 6. 금지 프롬프트 예시

아래와 같은 요청을 받더라도 바로 반영하지 않는다. 각 요청이 왜 위험한지 먼저 설명한 뒤 팀 확인을 받는다.

- `"필요 없어 보이는 필드를 정리해줘"` — 현재 사용되지 않아 보이는 필드도 UI나 저장 계층에서 의존하고 있을 수 있다. 팀 합의 없이 삭제하면 다른 역할의 작업이 중단된다.
- `"더 좋은 구조로 모델을 리팩터링해줘"` — `TopicDocument`는 팀 공용 계약이다. 구조 변경은 모든 계층에 동시 영향을 미치므로 단독으로 판단할 수 없다.
- `"UI에 맞게 필드를 조금 바꿔줘"` — UI는 `TopicDocument`를 읽는 쪽이다. UI 편의를 위해 도메인 계약을 바꾸는 것은 계층 분리 원칙에 어긋난다.
- `"DB 저장하기 쉽게 모델을 수정해줘"` — 저장 계층은 계약에 맞게 구현하는 쪽이다. 저장 편의를 위해 계약을 바꾸는 것은 의존 관계를 역전시킨다.
