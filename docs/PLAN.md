# Newspedia 계획

## 1. 목표

본 프로젝트는 **환각을 최소화하면서, 팀이 수집한 뉴스 및 문서 데이터 범위 안에서 답변하는 RAG 기반 질의응답 시스템**을 구축한다. 질문에 답하는 챗봇에 머무르지 않고, 뉴스 기사와 이슈 문서를 벡터 형태로 임베딩해 검색 가능한 지식 기반을 만들고, 사용자가 검색과 탐색을 함께 수행할 수 있는 서비스 형태로 구현한다.

이 프로젝트는 두 가지 사용자 경험을 하나로 결합한다.

첫째, 사용자가 질문을 입력하면 벡터 데이터베이스에서 관련 문서를 검색하고, LLM이 그 범위 안에서 답변하는 **RAG 기반 검색 경험**을 제공한다. 이때 답변은 반드시 근거가 되는 문서나 기사와 함께 제시되어야 하며, 사용자가 답변의 출처를 직접 확인할 수 있어야 한다.

둘째, 사용자가 질문을 하지 않더라도 주요 이슈를 카드 형태로 탐색하고, 선택한 이슈를 구조화된 문서로 읽을 수 있는 **문서형 탐색 경험**을 제공한다. 검색 중심 사용성과 문서 기반 탐색 사용성은 이 흐름 안에서 함께 제공한다.

이 서비스의 핵심 가치는 다음과 같다.

- **환각을 줄인 답변**: 검색된 내부 데이터 범위를 벗어나지 않는 답변을 생성한다.
- **출처가 있는 응답**: 답변의 근거가 되는 기사와 문서를 함께 보여준다.
- **검색과 탐색의 결합**: 상단 검색창으로 질문할 수도 있고, 하단 카드와 네비게이션 트리로 이슈를 탐색할 수도 있다.
- **지속적으로 갱신되는 이슈 문서**: 동일한 이슈를 매번 다시 질문하지 않아도, 구조화된 문서 형태로 다시 방문할 수 있다.

또한 문서 페이지의 읽기 경험은 나무위키와 같은 문서형 서비스가 제공하는 강점을 일부 참고한다. 여기서 참고하는 것은 협업 편집 구조가 아니라, 목차를 통해 섹션을 빠르게 이동하고, 관련 문서 링크를 따라가며 주제를 확장할 수 있는 탐색 경험이다. 본 서비스는 이를 **네비게이션 트리**, **문서 내부 목차**, **관련 이슈 연결**의 형태로 구현한다.

## 2. 주요 산출물

본 프로젝트의 산출물은 다음과 같다.

1. **수집된 데이터 및 데이터 전처리 문서**
   뉴스 기사 수집 방식, 사용한 데이터 소스, 정제 규칙, 중복 제거 기준, 저장 구조를 설명하는 문서이다.

2. **시스템 아키텍처 문서**
   RAG 파이프라인, 벡터 데이터베이스, LangChain 연동, Airflow 배치 구조, UI 흐름을 설명하는 문서이다.

3. **개발된 소프트웨어**
   RAG 기반 LLM과 벡터 데이터베이스를 연동하는 구현 코드, 이슈 문서 생성 체인, 검색 UI, 카드 UI, 상세 페이지를 포함한다.

4. **테스트 계획 및 결과 보고서**
   검색 품질, 환각 방지 기준, 답변 출처 표시, 임베딩 및 검색 정확도, UI 동작 검증 계획과 결과를 정리한 보고서이다.

5. **발표 자료**
   문제 정의, 목표, 시스템 구조, 구현 내용, 시연 흐름, 결과 및 한계를 정리한 자료이다.

## 3. 베타 범위

### 베타에서 구현하는 것

베타 시연에서는 다음 경험을 한 화면 안에서 자연스럽게 연결한다.

1. 화면 최상단에서 검색창에 질문을 입력한다.
2. 시스템이 벡터 데이터베이스에서 관련 기사 및 이슈 문서를 검색한다.
3. LLM이 검색 결과를 바탕으로 답변을 생성한다.
4. 답변 아래에 근거 기사와 관련 문서를 보여준다.
5. 사용자는 하단 카드 섹션에서 주요 이슈를 탐색할 수 있다.
6. 카드 클릭 시 상세 문서 페이지로 이동한다.
7. 상세 페이지에서는 목차와 네비게이션 트리를 통해 문서와 관련 이슈를 계속 탐색할 수 있다.

이를 위해 베타에서는 다음 기능을 구현한다.

- 뉴스 기사 수집 및 원본 저장
- 기사 본문 정제
- 기사 청크 임베딩 생성
- 벡터 데이터베이스 저장 및 검색
- LangChain 기반 Retriever + LLM 체인
- 구조화된 이슈 문서 생성
- 상단 검색창 기반 RAG 질의응답 UI
- 하단 이슈 카드 리스트 UI
- 네비게이션 트리와 목차를 포함한 문서 상세 화면

현재 저장소는 베타 전체 기능을 완성한 상태가 아니라, 팀이 병렬 개발을 시작할 수 있는 **초안 스캐폴드 상태**까지 정리되어 있다. 즉, Docker 기반 개발 및 서버 구동, Airflow 3 서비스 분리, DAG 등록, `IssueDocument` 계약, 문서 생성 체인 진입점, 데모 UI, 팀 운영 문서는 준비되어 있으며, 실제 수집 로직, 저장 로직, 임베딩, 검색, LangChain RAG 체인, 실데이터 조회는 역할 분담 후 채우는 것을 전제로 한다.

### 베타 이후 확장 계획

베타 이후에는 다음 기능을 순차적으로 추가한다.

- **보도 차이 분석**: 동일한 이슈를 다루는 기사 간 관점, 강조점, 추가 정보의 차이를 별도 섹션으로 제공한다.
- **페이지 내부 Q&A**: 특정 이슈 문서 페이지 내부에서만 추가 질문을 수행한다.
- **타임라인 뷰**: 이슈의 시간 흐름을 시각적으로 표현한다.
- **버전 히스토리**: 문서가 갱신될 때마다 이전 버전과의 차이를 비교할 수 있도록 한다.
- **카테고리 확장**: 경제, 정치 등 다중 카테고리를 지원한다.
- **사용자 기능**: 로그인, 맞춤화, 트렌드 대시보드 등을 추가한다.

## 4. 핵심 사용자 경험

### 4-1. 검색 중심 경험

메인 화면의 최상단에는 네이버나 구글과 유사한 형태의 큰 검색창을 배치한다. 사용자는 자연어 질문을 입력하고, 시스템은 관련 기사 청크와 이슈 문서를 벡터 검색으로 가져온 뒤 LLM이 이를 바탕으로 답변을 생성한다. 답변은 검색된 문서 범위 안에서만 이루어져야 하며, 답변 바로 아래에는 근거 기사와 관련 이슈 문서를 함께 표시한다.

이 검색 경험은 단순한 키워드 검색이 아니라, **RAG 기반 질의응답 인터페이스**여야 한다. 즉, 사용자는 "이 이슈의 핵심은 뭐야?", "최근 반도체 관련 주요 뉴스가 뭐야?", "이 기사와 관련된 다른 이슈는 뭐야?" 같은 질문을 할 수 있어야 한다.

### 4-2. 탐색 중심 경험

검색창 아래에는 카드 섹션을 배치한다. 여기에는 최근 생성된 주요 이슈, 중요 이슈, 자주 조회된 이슈 등을 카드 형태로 나열한다. 사용자는 질문을 하지 않더라도 카드만 훑어보면서 지금 어떤 이슈가 있는지 파악할 수 있어야 한다.

카드를 클릭하면 해당 이슈의 문서 상세 페이지로 이동한다. 이 페이지는 단순한 게시글이 아니라, 개요, 배경, 핵심 사실, 근거 기사, 관련 이슈를 구조적으로 보여 주는 문서형 페이지여야 한다.

### 4-3. 네비게이션 트리 경험

문서 상세 페이지에는 다음 세 가지 네비게이션 요소를 반영한다.

- **문서 내부 목차**: 개요, 배경, 핵심 사실, 근거 기사, 관련 이슈로 바로 이동할 수 있도록 한다.
- **관련 이슈 트리**: 현재 이슈와 연결된 다른 이슈를 트리 또는 연결 목록 형태로 보여준다.
- **주제 탐색 트리**: 상위 주제 → 하위 이슈 흐름으로 문서를 확장 탐색할 수 있도록 한다.

이 네비게이션 트리는 검색 중심 사용성과 문서형 탐색 경험을 이어 주는 핵심 요소다.

## 5. 이슈 문서 구조

하나의 이슈 문서는 본문 섹션과 네비게이션 요소로 구성한다. 전체 방향은 문서형 가독성과 기사 근거 중심의 신뢰성을 결합하는 데 둔다.

### 본문 섹션

**개요** 섹션은 이슈의 핵심 내용을 3~5문장으로 요약한다. 사용자는 이 섹션만 읽더라도 "무슨 일이 있었는지"를 파악할 수 있어야 한다. 구체적인 수치나 인물이 기사에 명시되어 있다면 포함하며, 추측이나 의견은 배제하고 사실 중심으로 작성한다.

**배경** 섹션은 해당 이슈를 이해하기 위한 맥락과 전사를 제공한다. 기사 본문에 명시된 배경 정보를 우선적으로 반영하되, LLM의 일반 지식을 활용한 보강도 허용한다. 다만 기사에 직접 근거하지 않은 내용은 반드시 `[일반 배경 정보]`로 표기하여, 기사 기반 정보와 일반 지식을 구분할 수 있도록 한다.

**핵심 사실** 섹션은 2개 이상의 기사가 공통으로 다루는 사실을 목록 형태로 정리한다. 복수 매체에서 공통으로 확인된 내용만을 포함하여, 사용자가 신뢰도 높은 사실을 빠르게 확인할 수 있도록 한다.

**근거 기사** 섹션은 문서 생성에 사용된 기사들의 메타데이터를 제공한다. 제목, 매체명, 발행일, 원문 URL을 포함하며, 사용자가 필요할 경우 원문을 직접 확인할 수 있도록 한다. 이 정보는 LLM이 생성하지 않고, 저장된 기사 데이터를 기반으로 결정적으로 조합한다.

### 네비게이션 요소

**목차**는 문서 상단에 배치하여 사용자가 원하는 섹션으로 바로 이동할 수 있도록 한다.

**관련 이슈**는 문서 하단에 배치하고, 현재 이슈와 유사한 다른 이슈 2~3개를 링크 형태로 제공한다.

**네비게이션 트리**는 문서 내부 목차와 관련 이슈 연결을 확장한 구조로, 사용자가 현재 이슈의 상위 주제와 하위 관련 이슈를 탐색할 수 있도록 한다.

### 데이터 계약: IssueDocument

시스템 전체에서 이슈 문서를 주고받을 때 사용하는 구조화된 데이터 계약은 다음과 같다. 이 모델은 분석 체인의 최종 출력이자 UI의 입력이며, 체인, 저장, UI는 모두 이 계약을 기준으로 동작한다.

```python
class SourceRef(BaseModel):
    article_id: int
    title: str
    publisher: str
    url: str
    published_at: datetime | None = None

class RelatedIssue(BaseModel):
    issue_id: int
    title: str

class IssueDocument(BaseModel):
    issue_id: int
    title: str
    overview: str
    background: str
    key_facts: list[str]
    source_articles: list[SourceRef]
    related_issues: list[RelatedIssue]
    generated_at: datetime
```

## 6. RAG 질의응답 구조

질의응답 흐름은 아래와 같다.

1. 사용자가 검색창에 질문을 입력한다.
2. 질의를 임베딩하거나 검색 가능한 형태로 변환한다.
3. 벡터 데이터베이스에서 관련 기사 청크와 이슈 문서를 검색한다.
4. LangChain 기반 Retriever가 검색 결과를 정리한다.
5. LLM이 검색 결과 범위 안에서 답변을 생성한다.
6. 답변과 함께 근거 기사와 관련 문서를 보여준다.

이 구조의 핵심은 "검색 없이 답하지 않는 것"이다. 검색 결과가 부족하면 답변을 과도하게 확장하지 않고, 부족하다고 명시하거나 근거 부족 상태를 드러내야 한다.

벡터 검색 대상은 두 층으로 나눌 수 있다.

- **기사 청크 검색**: 세부 사실 확인, 질문 응답용
- **이슈 문서 검색**: 상위 맥락 요약, 관련 주제 탐색용

베타에서는 우선 기사 청크 기반 검색을 중심으로 구현하고, 이슈 문서는 보조 검색 또는 탐색 결과로 연결하는 방식이 현실적이다.

## 7. 시스템 아키텍처

전체 데이터 흐름은 뉴스 수집에서 시작해 이슈 문서 생성과 RAG 응답, UI 표시로 이어진다.

```text
[News Search APIs]
   ├─ Primary: Naver News API
   ├─ Fallback: Brave Search API
   └─ Assist: Tavily Search/Extract
            │
            ▼
        [MongoDB]
   원본 응답 저장 및 상태 관리
            │
            ▼
      [Preprocessing]
   본문 스크래핑 / 정제 / 중복 판단
            │
            ▼
 [PostgreSQL + pgvector]
 정제 기사 / 청크 / 임베딩 / 이슈 / 문서
      │                    │
      │                    └─ [Issue Document Generator]
      │                       개요 / 배경 / 핵심 사실 생성
      ▼
 [Retriever + LangChain]
 질문 → 검색 결과 조합 → LLM 답변
      │
      ├─ 근거 기사
      ├─ 관련 문서
      └─ 관련 이슈 트리
            │
            ▼
     [Streamlit UI]
 상단 검색창 / 답변 / 카드 섹션 / 상세 문서
```

뉴스 검색 API에서 기사 목록을 가져오면, 원본 응답을 MongoDB에 그대로 저장한다. 이후 기사 본문을 스크래핑하고 정제한 뒤 PostgreSQL과 pgvector에 저장한다. 정제된 기사는 텍스트 청크로 분할되고, 각 청크는 임베딩 벡터로 변환되어 검색 가능한 형태로 저장된다.

이후 시스템은 두 갈래로 동작한다. 첫째는 이슈 클러스터링과 문서 생성 흐름이다. 유사 기사들을 하나의 이슈로 묶고, 각 이슈를 `IssueDocument`로 생성해 문서 페이지의 기반으로 사용한다. 둘째는 사용자의 검색 질문에 응답하는 RAG 흐름이다. Retriever가 기사 청크와 문서를 검색하고, LangChain 체인이 이를 조합하여 답변을 생성한다.

## 8. 저장 구조

데이터베이스를 MongoDB와 PostgreSQL로 분리하는 이유는 저장 대상과 활용 방식이 다르기 때문이다.

### MongoDB

MongoDB는 뉴스 API 원본 응답을 그대로 저장한다. provider 정보, fallback 사용 여부, 수집 상태 필드를 함께 기록한다.

### PostgreSQL + pgvector

PostgreSQL은 정제된 기사와 관계형 메타데이터를 저장하고, pgvector는 임베딩 검색을 담당한다.

목표 테이블 구성은 다음과 같다.

- `articles`
- `article_chunks`
- `article_embeddings`
- `issues`
- `issue_articles`
- `issue_documents`
- `document_sections`
- `document_source_refs`

이 중 `article_chunks`와 `article_embeddings`는 RAG 검색에 직접 사용되고, `issues`와 `issue_documents`는 카드 UI와 문서 상세 페이지에 직접 사용된다.

## 9. 배치 파이프라인

전체 파이프라인은 4개의 Airflow DAG로 구성하며, 뉴스 수집부터 이슈 문서 생성까지를 자동 수행한다.

**collect_news**는 기사 목록을 가져와 원본 JSON 응답을 MongoDB에 저장한다.

**prepare_articles**는 본문을 스크래핑하고 정제한 뒤 PostgreSQL의 `articles` 테이블에 저장한다.

**embed_articles**는 기사 청크를 생성하고 임베딩 벡터를 만든 뒤, 이슈 클러스터링에 필요한 데이터를 저장한다.

**analyze_issues**는 각 이슈에 속한 기사들을 읽고, `IssueDocument`를 생성해 데이터베이스에 저장한다.

현재 저장소 초안에서는 위 4개 DAG와 `src/pipeline` 진입점이 모두 존재하며, DAG는 Airflow에서 실제로 등록되는 상태까지 준비되어 있다. 다만 각 단계의 세부 비즈니스 로직은 아직 스캐폴드 수준이고, 현재 DAG는 수동 검증을 우선하기 위해 `schedule=None`으로 등록되어 있다. 실제 주기 실행 정책은 Airflow 담당자가 통합 단계에서 확정하고 반영한다.

## 10. UI 구성

사용자 화면은 크게 세 영역으로 구성한다.

### 10-1. 상단 검색 영역

화면 최상단에는 검색엔진 스타일의 큰 검색창을 둔다. 사용자가 질문을 입력하면 답변과 근거 문서를 바로 보여준다.

답변 영역에는 다음 요소를 포함한다.

- 질문
- RAG 기반 답변
- 근거 기사 목록
- 관련 문서 또는 관련 이슈 링크

### 10-2. 카드 섹션

검색 영역 아래에는 자동 생성된 주요 이슈를 카드 형태로 나열한다. 각 카드에는 이슈 제목, 한 줄 개요, 최근 갱신 시각, 포함 기사 수를 표시한다.

### 10-3. 상세 문서 페이지

카드를 선택하면 해당 이슈의 문서 상세 페이지로 이동한다. 이 페이지에는 다음 요소를 포함한다.

- 목차
- 개요
- 배경
- 핵심 사실
- 근거 기사
- 관련 이슈
- 네비게이션 트리

현재 UI는 실데이터 대신 데모 `IssueDocument`를 기반으로 동작한다. 우선 레이아웃과 흐름부터 확정해 둔 상태이며, 이후 저장 계층과 검색 계층이 준비되면 동일한 데이터 계약을 유지한 채 실데이터로 전환한다.
